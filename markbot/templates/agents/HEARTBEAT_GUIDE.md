# 心跳行为指南

## 决策流程

收到心跳轮询时，按以下顺序判断：

```
1. HEARTBEAT.md 有任务？ → 执行任务
2. 安静时段（23:00-08:00）？ → HEARTBEAT_OK
3. 距上次主动联系 <2h？ → HEARTBEAT_OK
4. 否则 → 执行一项主动检查
```

## 主动检查项（轮流执行）

每天 2-4 次，轮流检查以下项目：

- **邮件** — 有紧急未读消息？
- **日历** — 未来 24-48h 有事件？
- **提及** — 社交通知？
- **天气** — 用户可能出门时相关？

## 状态追踪

在 `memory/heartbeat-state.json` 中记录检查时间：

```json
{
  "lastChecks": {
    "email": 1703275200,
    "calendar": 1703260800,
    "weather": null
  }
}
```

## 何时主动联系用户

- 重要邮件到达
- 日历事件即将开始（<2h）
- 发现有趣的东西
- 距上次交流 >8h

## 何时保持安静

- 深夜（23:00-08:00），除非紧急
- 用户明显在忙
- 上次检查后没有新内容
- 距上次检查 <30 分钟

## 可自主执行的工作

- 读取和整理记忆文件
- 检查项目状态（git status 等）
- 更新文档
- 提交和推送自己的更改
- 审查和更新 `MEMORY.md`

## 记忆维护

每隔几天，用一次心跳来：

1. 读取最近的 `memory/YYYY-MM-DD.md` 文件
2. 识别值得长期保留的重要事件、教训或洞察
3. 更新 `MEMORY.md` 中的精炼内容
4. 删除 `MEMORY.md` 中不再相关的过时信息

每日文件是原始笔记，`MEMORY.md` 是精炼智慧。

## 心跳 vs Cron

| 场景 | 用心跳 | 用 cron |
|------|--------|---------|
| 多项检查批量执行 | ✅ | |
| 需要对话上下文 | ✅ | |
| 时间可以略有偏差 | ✅ | |
| 精确定时（每周一 9:00） | | ✅ |
| 任务需要独立于主会话 | | ✅ |
| 一次性提醒 | | ✅ |
| 需要用不同模型 | | ✅ |

**提示：** 将类似的定期检查合并到 `HEARTBEAT.md`，而不是创建多个 cron 任务。
